1. บริษัทประกันภัยข้ามชาติแห่งหนึ่งเก็บข้อมูลลูกค้าแยกตามประเภทกรมธรรม์ (ชีวิต, รถยนต์, สุขภาพ)
ในรูปแบบแฟ้มข้อมูล (File Processing System) โดยแต่ละแผนกพัฒนาแอปพลิเคชันขึ้นมาใช้งานเอง
ต่อมาพบว่าเมื่อลูกค้าย้ายที่อยู่ ฝ่ายประกันชีวิตแก้ไขข้อมูลแล้ว แต่ฝ่ายรถยนต์ยังคงส่งไปที่อยู่เดิม
นอกจากนี้ระบบยังไม่รองรับการพ่วงสัญญาข้ามแผนก เพราะโครงสร้างข้อมูลถูกกำหนดไว้ในโปรแกรม
หากพิจารณาตามหลัก Data Independence ปัญหาด้านโครงสร้างนี้ส่งผลกระทบที่รุนแรงที่สุดในด้านใด

ก. ความยากลำบากในการแก้ไขโครงสร้างข้อมูล (Structural Dependency)

ข. การละเมิดกฎความปลอดภัยของข้อมูล (Security Violation)

ค. การเกิดความซ้ำซ้อนของข้อมูลที่ควบคุมไม่ได้ (Uncontrolled Redundancy)

ง. ข้อจำกัดด้านความจุของอุปกรณ์จัดเก็บข้อมูล (Storage Limitation)

2. ในระบบฐานข้อมูลธนาคารที่รองรับธุรกรรมพร้อมกันหลายหมื่นรายการต่อวินาที DBMS
ต้องทำหน้าที่เป็นตัวกลางที่สำคัญยิ่ง สมมติว่าในจังหวะที่ระบบกำลังหักเงินจากบัญชีต้นทาง
เพื่อโอนไปยังปลายทาง แล้วเกิดไฟฟ้าขัดข้องทำให้เครื่องเซิร์ฟเวอร์หยุดทำงานลงทันที
หน้าที่ใดของ DBMS ที่ถูกออกแบบมาเพื่อจัดการความสมบูรณ์ของข้อมูลในสภาวะวิกฤต
เพื่อให้ 'ยอดเงินไม่หายและไม่เพิ่มขึ้นผิดปกติ' จากเหตุการณ์ขัดข้องดังกล่าว

ก. การสร้างมุมมองข้อมูลที่แตกต่างกันสำหรับผู้ใช้ (Data View Management)

ข. การควบคุมความถูกต้องและความปลอดภัย (Integrity and Security Control)

ค. การทำหน้าที่เป็นอิสระจากส่วนของฮาร์ดแวร์ (Hardware Independence)

ง. การลดความซ้ำซ้อนของข้อมูล (Redundancy Reduction)

3. มหาวิทยาลัยแห่งหนึ่งใช้งานระบบฐานข้อมูลกลาง (Centralized Database) หากผู้ดูแลระบบ
ตัดสินใจเปลี่ยนโครงสร้างการเก็บ 'รหัสนิสิต' จากตัวเลข 8 หลัก เป็นตัวอักษรผสมเลข 10 หลัก
เพื่อรองรับการขยายตัว หากระบบนี้มีประสิทธิภาพตามหลักการของฐานข้อมูลที่ถูกต้อง
ข้อใดคือผลกระทบที่ควรจะเกิดขึ้นกับ 'โปรแกรมลงทะเบียน' ที่นิสิตใช้งานอยู่ในปัจจุบัน

ก. โปรแกรมไม่ต้องถูกแก้ไขโค้ดเนื่องจากมีความเป็นอิสระทางตรรกะ (Logical Data Independence)

ข. ความเร็วในการประมวลผลจะลดลงกึ่งหนึ่งเนื่องจากจำนวนบิตในระดับ Hardware เพิ่มขึ้น

ค. ต้องทำการเขียนโปรแกรมลงทะเบียนขึ้นใหม่ทั้งหมดเนื่องจากชนิดข้อมูล (Data Type) เปลี่ยนไป

ง. ข้อมูลเดิมที่เป็นตัวเลข 8 หลักจะถูกลบทิ้งโดยอัตโนมัติเพราะไม่ตรงกับโครงสร้างใหม่

4. ในการออกแบบระบบฐานข้อมูลโรงพยาบาล 'นายแพทย์วิชัย' ต้องการดูประวัติรักษาในรูปแบบกราฟ
ขณะที่ 'เจ้าหน้าที่การเงิน' ต้องการดูเฉพาะรายการค่าใช้จ่ายเพื่อออกใบเสร็จรับเงินให้แก่คนไข้
องค์ประกอบด้านบุคลากรในข้อใดที่มีหน้าที่ออกแบบและกำหนด 'มุมมอง (View)' ที่แตกต่างกัน
เพื่อให้ทั้งสองฝ่ายเข้าถึงข้อมูลได้อย่างเหมาะสมและปลอดภัยตามขอบเขตงานของตนเอง

ก. ผู้ใช้ระดับสูง

ข. โปรแกรมเมอร์

ค. พนักงานบันทึกข้อมูล

ง. ผู้จัดการฐานข้อมูล

5. ข้อมูลในคอมพิวเตอร์ถูกเก็บเป็นเลขฐานสอง (Bit) โดย 8 บิต รวมเป็น 1 ไบต์ (แทน 1 ตัวอักษร)
และเมื่อนำตัวอักษรหลายตัวมาต่อกันจะกลายเป็นฟิลด์ หากระบบต้องการเก็บคำว่า 'DATABASE'
ซึ่งมีทั้งหมด 8 ตัวอักษร จะต้องใช้หน่วยความจำกี่บิต และคำนี้จัดเป็นหน่วยข้อมูลในระดับใด
ตามลำดับชั้นการจัดเก็บข้อมูล (Data Hierarchy) ที่กำหนดไว้ข้างต้น

ก. 64 บิต และจัดเป็นระดับ Byte

ข. 8 บิต และจัดเป็นระดับ Record

ค. 512 บิต และจัดเป็นระดับ File

ง. 64 บิต และจัดเป็นระดับ Field

6. เจ้าหน้าที่บริษัทบ่นว่า 'เมื่อเราต้องการเปลี่ยนลำดับการเก็บข้อมูลจาก ชื่อ-นามสกุล เป็น
นามสกุล-ชื่อ เราต้องแจ้งให้โปรแกรมเมอร์มาแก้ไขโปรแกรมทุกตัวที่เรียกใช้งานไฟล์นี้'
สถานการณ์นี้แสดงให้เห็นว่าระบบเดิมขาดคุณสมบัติในข้อใด และระบบฐานข้อมูล (DBMS)
จะสามารถเข้ามาแก้ไขปัญหาดังกล่าวนี้ได้อย่างไรเพื่อให้เกิดประสิทธิภาพสูงสุดแก่บริษัท

ก. ขาดข้อมูลภายในองค์กร และระบบฐานข้อมูลจะช่วยดึงข้อมูลจากภายนอก

ข. ขาดอิสระของข้อมูล และแยกโครงสร้างออกจากโปรแกรม

ค. ขาดความปลอดภัย และระบบฐานข้อมูลจะช่วยใส่รหัสผ่านให้ไฟล์

ง. ขาดความซ้ำซ้อนของข้อมูล (Data Redundancy) และระบบฐานข้อมูลจะช่วยสร้างไฟล์เพิ่ม

7. บริษัทประกันภัยเปลี่ยนระบบจัดเก็บจาก Hard Disk เป็น SSD และใช้อัลกอริทึมบีบอัดใหม่
ทำให้ Internal Schema เปลี่ยนแปลงอย่างมาก แต่โปรแกรมในระดับ External Level
ยังคงทำงานได้ปกติโดยไม่ต้องแก้ไขโค้ด สถานการณ์นี้สะท้อนถึงคุณสมบัติในข้อใดชัดเจนที่สุด
และกระบวนการใดในระบบที่ต้องทำงานเพื่อรองรับการเปลี่ยนแปลงในระดับโครงสร้างนี้

ก. Logical Data Independence และเกิดการปรับปรุง External/Conceptual Mapping

ข. User View Optimization และเกิดการปรับปรุงเฉพาะระดับ External Schema

ค. Physical Data Independence และเกิดการปรับปรุง Conceptual/Internal Mapping

ง. Data Redundancy Control และเกิดการปรับปรุงระดับโครงสร้างตาราง

8. มหาวิทยาลัยเพิ่มฟิลด์ 'คะแนนกิจกรรมสะสม' ในระดับ Conceptual Schema ของฐานข้อมูล
แต่โปรแกรมฝ่ายธุรการที่มองผ่าน External Schema (เห็นเฉพาะ รหัส ชื่อ สาขา) ยังคงทำงาน
ได้ปกติโดยไม่ต้องแก้ไขโค้ด แม้จะมีการเพิ่มข้อมูลใหม่ลงไปในโครงสร้างหลักของระบบแล้วก็ตาม
สถานการณ์นี้แสดงถึงคุณสมบัติในด้านใด และเป็นเพราะเหตุใดโปรแกรมเดิมจึงยังใช้งานได้

ก. ความเป็นอิสระของข้อมูลทางตรรกะ เนื่องจากเปลี่ยนระดับแนวคิดได้โดยไม่กระทบระดับภายนอก

ข. การใช้ Subschema ในระดับภายใน เพื่อซ่อนฟิลด์ใหม่ไม่ให้ระบบประมวลผลเห็น

ค. ความเป็นอิสระของข้อมูลทางกายภาพ (Physical Data Independence) เนื่องจากไม่มีการเปลี่ยนตำแหน่งไฟล์ในดิสก์

ง. การทำ Data Integrity Mapping เพื่อตรวจสอบความถูกต้องของคะแนนสะสมก่อนแสดงผล

9. บริษัทเกมต้องการเก็บข้อมูล 'ตัวละคร' ที่มีทั้งสถานะ (เช่น เลือด, พลังเวท) และพฤติกรรม
(เช่น การเดิน, การโจมตี, การคำนวณความเสียหาย) จากความต้องการที่ซับซ้อนข้างต้นนี้
แบบจำลองข้อมูลประเภทใดเหมาะสมที่สุด เพื่อให้นักพัฒนาเขียนโปรแกรมโต้ตอบข้อมูล
ได้อย่างเป็นธรรมชาติ และสามารถลดความซับซ้อนในการ Mapping ข้อมูลในเชิงโครงสร้าง

ก. แบบจำลองข้อมูลแบบแบนราบ (Flat File Data Model)

ข. แบบจำลองข้อมูลเชิงวัตถุ (Object-Oriented Data Model)

ค. แบบจำลองข้อมูลเชิงตรรกะ (Logical Data Model) แบบเดิม

ง. แบบจำลองข้อมูลแบบลำดับชั้น (Hierarchical Data Model)

10. หากท่านเป็น DBA และต้องการเปลี่ยนวิธีการจัดเก็บข้อมูลในฮาร์ดดิสก์จากแบบเรียงลำดับ
(Sequential) เป็นแบบดัชนี (Indexing) โดยที่ชื่อฟิลด์และความสัมพันธ์เดิมไม่มีการเปลี่ยนแปลง
นักศึกษาต้องเข้าไปจัดการที่สคีมาในระดับใด และเหตุการณ์นี้จะส่งผลกระทบอย่างไรต่อโปรแกรม
ประยุกต์ของผู้ใช้ที่เชื่อมต่อกับระบบฐานข้อมูลตามหลักการ Data Independence

ก. จัดการที่ Conceptual Schema และทำให้ผู้ใช้มองเห็นดัชนีปรากฏบนหน้าจอโปรแกรม

ข. จัดการทุกระดับพร้อมกัน เพราะการเพิ่มดัชนีคือการเปลี่ยนความหมายเชิงตรรกะของข้อมูล

ค. จัดการที่ External Schema และทำให้โปรแกรมต้องเขียนใหม่เพื่อรองรับระบบดัชนี

ง. จัดการที่ Internal Schema และไม่ส่งผลกระทบใดๆ ต่อโปรแกรมประยุกต์ของผู้ใช้

11. วิศวกรพบว่าในระบบแบบลำดับชั้น (Hierarchical) เมื่อทำการ 'ลบข้อมูลฝ่ายขาย' ออกจากระบบ
ข้อมูลพนักงานที่สังกัดฝ่ายนั้นจะหายไปด้วยโดยอัตโนมัติ (Cascading Delete) จากสถานการณ์นี้
ข้อใดอธิบายเหตุผลทางสถาปัตยกรรมได้ถูกต้อง และปัญหานี้ถูกแก้ไขอย่างไรในแบบจำลองเชิงสัมพันธ์
เพื่อให้ข้อมูลพนักงานยังคงอยู่ได้แม้จะมีการเปลี่ยนแปลงโครงสร้างของหน่วยงานต้นสังกัด

ก. เป็นเพราะโครงสร้างแบบ Root และ Child ซึ่งลูกอยู่เดี่ยวๆ ไม่ได้; แก้ไขโดยใช้ตารางแยกและเชื่อมด้วย Foreign Key

ข. เป็นเพราะหน่วยจัดเก็บในสมัยก่อนมีขนาดเล็ก; แก้ไขโดยการเพิ่มพื้นที่จัดเก็บให้พนักงานอยู่ได้โดยไม่ต้องสังกัดฝ่าย

ค. เป็นเพราะแบบลำดับชั้นไม่มีระดับ Conceptual Level; แก้ไขโดยการเพิ่มระดับแนวคิดเข้าไปในระบบตาราง

ง. เป็นเพราะการกระทำ Mapping ระหว่างระดับ External/Internal ไม่สมบูรณ์; แก้ไขโดยการใช้ Object-Oriented มาแทน

12. คำว่า 'สคีมา (Schema)' ถูกเปรียบเทียบว่าเป็นเค้าร่างหรือรูปแบบ หากเราพิจารณาฐานข้อมูลขนาดใหญ่ ที่มีกลุ่มผู้ใช้หลากหลายประเภท เช่น ผู้ใช้ทั่วไป, ผู้ดูแลระบบ (DBA) และนักพัฒนาโปรแกรมเมอร์
ข้อใดคือคำอธิบายความสัมพันธ์ของสคีมาในระดับต่างๆ ได้ถูกต้องและชัดเจนที่สุดตามหลักทฤษฎี
สถาปัตยกรรมระบบฐานข้อมูล 3 ระดับ (Three-Level Architecture)

ก. มี Internal Schema หลายชุดเพื่อกระจายข้อมูลไปยังฮาร์ดดิสก์หลายลูก แต่มี Conceptual Schema ชุดเดียว

ข. สคีมาเป็นเพียงภาพวาดในกระดาษสำหรับผู้ออกแบบ ไม่เกี่ยวข้องกับการทำงานจริงของซอฟต์แวร์ DBMS

ค. มี Conceptual Schema เพียงหนึ่งเดียวต่อหนึ่งฐานข้อมูล แต่มี External Schema ได้หลายชุดตามกลุ่มผู้ใช้

ง. ทุกระดับจะมีสคีมาเพียงชุดเดียวเท่านั้น เพื่อป้องกันความสับสนและลดความซับซ้อนในการทำ Mapping

13. หากบริษัทอสังหาริมทรัพย์ต้องการเก็บข้อมูล 'พิมพ์เขียว 3 มิติ' รวมถึงบันทึก 'ขั้นตอนการคำนวณราคา' ตามวัสดุที่เปลี่ยนไปโดยอัตโนมัติภายในตัวฐานข้อมูลเอง แบบจำลองข้อมูล (Data Model)
ในข้อใด จะช่วยลดภาระในการเขียนโปรแกรมประยุกต์ (Application Logic) ได้มากที่สุดในเชิงโครงสร้าง และการจัดการข้อมูลที่มีความซับซ้อนสูงตามความต้องการข้างต้นนี้

ก. Relational Data Model เพราะสามารถใช้ตารางเก็บพิกัด 3 มิติ และใช้ SQL คำนวณราคาได้เร็วกว่า

ข. Object-Oriented Data Model เพราะรองรับข้อมูลซับซ้อนและเก็บพฤติกรรมไว้กับข้อมูลได้

ค. Network Data Model เพราะพิมพ์เขียวมีความเชื่อมโยงของโครงสร้างกันแบบใยแมงมุม

ง. Hierarchical Data Model เพราะบ้านถูกแบ่งเป็นชั้นๆ และห้องต่างๆ ตามลำดับความสำคัญ

14. ทีมวิจัยพบว่าทุกครั้งที่มีการร้องขอข้อมูล ระบบต้องเสียเวลา (Overhead) ในการแปลคำสั่ง
ผ่านชั้นต่างๆ ถึง 2 ครั้ง แต่สถาปัตยกรรม ANSI-SPARC ยังคงเป็นมาตรฐานที่ยอมรับสูงสุด
นักศึกษาคิดว่าเหตุผลข้อใดที่สามารถหักล้างข้อเสียเรื่องความช้าในการแปลคำสั่ง (Mapping)
นี้ได้ดีที่สุดในบริบทของการดำเนินธุรกิจจริงที่มีการเปลี่ยนแปลงทางเทคโนโลยีอยู่ตลอดเวลา

ก. ความคุ้มค่าจากการไม่ต้องแก้ไขและคอมไพล์โปรแกรมใหม่ทั้งหมดเมื่อมีการเปลี่ยนโครงสร้างระดับล่าง

ข. การทำ Mapping ช่วยบีบอัดข้อมูลในระหว่างทางทำให้การส่งข้อมูลผ่านเครือข่ายทำได้รวดเร็วขึ้น

ค. ความเร็วของหน่วยประมวลผลเพิ่มขึ้นมากจนการ Mapping ไม่ส่งผลต่อประสิทธิภาพที่มนุษย์สัมผัสได้

ง. ระบบฐานข้อมูลสมัยใหม่ได้ยุบรวมระดับแนวคิดและระดับภายในเข้าด้วยกันเพื่อลด Overhead อยู่แล้ว

ท่านควรดำเนินการจัดการกับโครงสร้างข้อมูลและ Attribute ดังกล่าวอย่างไรจึงจะเหมาะสมที่สุด

เนื่องจากการอัปเดตรหัสแพทย์ในตารางคนไข้ไม่ครบทุกแถว สถานการณ์นี้สะท้อนถึงความล้มเหลว
ในการจัดการคำศัพท์ฐานข้อมูลตัวใดมากที่สุด และนักศึกษาควรใช้กลไกใดมาควบคุมเพื่อบังคับ
ไม่ให้เกิดปัญหาความไม่สอดคล้องของข้อมูลดังกล่าวภายในระบบฐานข้อมูลของโรงพยาบาล

ก. ความล้มเหลวในการใช้คีย์รอง (Secondary Key) และต้องใช้การเพิ่มจำนวน Tuple ในตารางหมอ

ข. ความล้มเหลวในการจัดการ Foreign Key และต้องใช้กฎ Referential Integrity ในการควบคุม

ค. ความล้มเหลวในการกำหนด Domain ของข้อมูล และต้องใช้การกำหนด Cardinality แบบ Many-to-Many

ง. ความล้มเหลวในการกำหนด Primary Key และต้องใช้การเพิ่ม Degree ของ Entity เพื่อเก็บข้อมูลเพิ่ม

หรือค่าที่ซ้ำกันได้ การกระทำนี้ถือเป็นการละเมิดกฎเหล็กของ 'Primary Key' ในด้านใด
และจะส่งผลกระทบต่อความถูกต้องของคำศัพท์ตัวใดในเชิงปริมาณข้อมูลในระบบ

ก. ละเมิดกฎ Domain Integrity และส่งผลกระทบต่อการนับจำนวน Degree ของตาราง

ข. ละเมิดกฎความสัมพันธ์แบบ 1:N และส่งผลกระทบต่อพื้นที่ว่างในหน่วยความจำ (Storage)

ค. ละเมิดกฎ Entity Integrity และส่งผลกระทบต่อความถูกต้องของ Cardinality ในระบบ

ง. ละเมิดกฎการสืบทอดข้อมูล และส่งผลกระทบต่อการสร้าง Foreign Key ในตารางพนักงาน

18. พิจารณาสถานการณ์การออกแบบฐานข้อมูลของบริษัทประกันภัยดังนี้ :
• ผู้เอาประกัน 1 คน สามารถถือครองกรมธรรม์ได้หลายฉบับ
• กรมธรรม์ 1 ฉบับ จะระบุชื่อผู้เอาประกันได้เพียงคนเดียวเท่านั้น
หากทีมออกแบบนำ Primary Key ของตารางผู้เอาประกัน ไปวางไว้ในตารางกรมธรรม์
การกระทำในลักษณะนี้เรียกว่าการสร้างคีย์ประเภทใด และสอดคล้องกับรูปแบบ
Cardinality Ratio แบบใดมากที่สุดตามหลักการของฐานข้อมูลเชิงสัมพันธ์

ก. สร้าง Foreign Key และสอดคล้องกับความสัมพันธ์แบบ One-to-Many (1:M)

ข. สร้าง Secondary Key และสอดคล้องกับความสัมพันธ์แบบ One-to-One (1:1)

ค. สร้าง Primary Key ร่วม (Composite Key) และสอดคล้องกับความสัมพันธ์แบบ Many-to-Many (M:N)

ง. สร้าง Domain Key และสอดคล้องกับความสัมพันธ์แบบ Recursive

19. พิจารณา Attribute 'Salary' ในตารางพนักงาน หากมีการกำหนดค่าข้อมูลไว้ได้ตั้งแต่
15,000 ถึง 100,000 บาทเท่านั้น ข้อจำกัดในลักษณะนี้ตามเนื้อหาบทที่ 3 เรียกว่าอะไร
และหากพนักงานบางคนยังไม่มีการระบุเงินเดือนเนื่องจากอยู่ในระหว่างช่วงทดลองงาน
ระบบควรยอมให้ช่องข้อมูลนี้มีค่าเป็นอะไรเพื่อให้ถูกต้องตามหลักการบริหารจัดการข้อมูล

ก. เรียกว่า Tuple และควรยอมให้ป้อนตัวเลขใดก็ได้ไปก่อนแล้วค่อยกลับมาแก้ไขในภายหลัง

ข. เรียกว่า Domain และควรยอมให้เป็นค่า NULL หากไม่ขัดต่อกฎเกณฑ์ของ Primary Key

ค. เรียกว่า Degree และควรยอมให้ใส่ข้อความ 'Pending' เพื่อเป็นการสื่อสารกับผู้ใช้งาน

ง. เรียกว่า Cardinality และควรยอมให้เป็นค่าศูนย์ (0) เพื่อให้ระบบสามารถคำนวณได้ง่าย

20. บริษัทขนส่งออกแบบฐานข้อมูล 'พัสดุ' (Package) และ 'สถานะ' (Status) โดยมีเงื่อนไขคือ
พัสดุ 1 ชิ้นมีได้หลายสถานะตามเวลา และสถานะ 1 อย่างสามารถใช้กับพัสดุได้หลายชิ้น
หากต้องการเก็บข้อมูล 'เวลาที่เปลี่ยนสถานะ' (Timestamp) เพิ่มเติมลงในระบบฐานข้อมูล
ท่านจะวิเคราะห์ Cardinality และโครงสร้างการจัดเก็บข้อมูลนี้ได้อย่างไรจึงจะถูกต้องที่สุด

ก. เป็นความสัมพันธ์แบบ One-to-One (1:1) เพราะพัสดุหนึ่งชิ้นมีได้เพียงหนึ่งสถานะ ณ เวลาใดเวลาหนึ่ง

ข. เป็นความสัมพันธ์แบบ One-to-Many (1:N) โดยเลือกเก็บ Timestamp ไว้ในตารางพัสดุเพื่อประหยัดพื้นที่

ค. เป็นความสัมพันธ์แบบ Recursive เพราะสถานะเก่าจะส่งต่อข้อมูลให้สถานะใหม่ภายในเอนทิตี้เดียวกันเสมอ

ง. เป็นความสัมพันธ์แบบ Many-to-Many (M:N) และต้องเก็บ Timestamp ไว้ในตารางกลางที่สร้างขึ้นใหม่

21. สมมติว่าท่านลบข้อมูล 'แผนกการตลาด' ออกจากตาราง DEPARTMENTS แต่ในตารางพนักงาน

ก. เกิดปัญหา Referential Integrity เนื่องจากมีคีย์นอกที่ชี้ไปยังข้อมูลที่ไม่มีอยู่จริงในตารางหลัก

ข. เกิดปัญหา Cardinality Explosion เนื่องจากพนักงานจะเพิ่มจำนวนขึ้นเพื่อชดเชยแผนกที่หายไป

ค. เกิดปัญหา Domain Overflow เนื่องจากค่าของรหัสแผนกเดิมเกินกว่าขอบเขตที่กำหนดไว้ในระบบ

ง. เกิดปัญหา Degree Mismatch เนื่องจากจำนวนคอลัมน์ในสองตารางไม่เท่ากันหลังจากทำการลบข้อมูล

22. คำว่า 'ดีกรี' (Degree) และ 'คาร์ดินัลลิตี้' (Cardinality) เป็นตัววัดเชิงปริมาณของรีเลชั่น

เหตุการณ์นี้แสดงว่าระบบใช้กลไก Referential Integrity ในรูปแบบใด และหากต้องการ
ให้รหัสในตารางลงทะเบียนเปลี่ยนตามโดยอัตโนมัติ ต้องกำหนดเงื่อนไขอย่างไรจึงจะเหมาะสม

ก. ปัจจุบันใช้รูปแบบ NO ACTION; และหากต้องการให้เปลี่ยนตามต้องตั้งค่าเป็น ON DELETE CASCADE

ข. ปัจจุบันใช้รูปแบบ DEFAULT; และหากต้องการให้เปลี่ยนตามต้องใช้กฎ Entity Integrity เข้าช่วยในการจัดการ

ค. ปัจจุบันใช้รูปแบบ SET NULL; และหากต้องการให้เปลี่ยนตามต้องตั้งค่าเป็น ON UPDATE RESTRICT

ง. ปัจจุบันใช้รูปแบบ RESTRICT; และหากต้องการให้เปลี่ยนตามต้องตั้งค่าเป็น ON UPDATE CASCADE

26. ในรีเลชั่น 'Students' กำหนดให้ ID เป็น Primary Key และ Email เป็น Unique Constraint
โดยระบบยอมให้ Email เป็น NULL ได้ในกรณีที่นักศึกษายังไม่มีข้อมูลอีเมล ณ ขณะที่สมัครเรียน
สถานการณ์นี้สะท้อนถึงความแตกต่างของกฎความถูกต้อง (Integrity Rule) ในข้อใดชัดเจนที่สุด
และหากมีนักศึกษาคนที่สองที่ไม่มีอีเมลเข้ามาเพิ่มอีก ระบบจะยอมให้ค่าเป็น NULL ได้หรือไม่

ก. ID ติดกฎ Domain Integrity; และระบบจะบังคับให้ใส่ค่าว่าง (Empty String) แทนค่า NULL เสมอ

ข. ID ติดกฎ Entity Integrity; และระบบยอมให้มี NULL ในช่อง Email ได้หลายครั้งเพราะ NULL ไม่ใช่ค่าซ้ำ

ค. ทั้งคู่ติดกฎ Referential Integrity; และระบบจะไม่อนุญาตให้เป็น NULL อีกเพราะจะผิดกฎการซ้ำซ้อน

ง. ทั้งคู่ติดกฎ Business Logic Integrity; และระบบจะทำการลบข้อมูลทิ้งโดยอัตโนมัติหากไม่มีการระบุค่า

27. ในตาราง 'พนักงาน' กำหนดให้ 'เลขที่บัตรประชาชน' เป็น Primary Key ต่อมาพบพนักงาน
ชาวต่างชาติที่ไม่มีเลขดังกล่าว ระบบจึงปฏิเสธการเพิ่มข้อมูล แม้จะพยายามเว้นว่างไว้ก็ตาม
เหตุการณ์ที่ระบบไม่ยอมให้เว้นว่างข้อมูลในส่วนที่เป็นคีย์หลักนี้ สะท้อนถึงการบังคับใช้กฎ
ความถูกต้องของข้อมูล (Integrity Rule) ในรูปแบบใดที่สำคัญที่สุดในระบบฐานข้อมูล

ก. Entity Integrity: ควบคุมความไม่ซ้ำและบังคับห้ามเป็นค่าว่าง (NOT NULL) สำหรับคีย์หลัก

ข. Domain Integrity: ควบคุมขอบเขตและชนิดของข้อมูลให้ตรงตามประเภทที่กำหนดไว้ในระบบ

ค. User-defined Integrity: กฎเกณฑ์ที่ผู้ใช้งานหรือผู้ออกแบบกำหนดขึ้นเองเพิ่มเติมตามเงื่อนไขธุรกิจ

ง. Referential Integrity: ควบคุมความถูกต้องของการอ้างอิงข้อมูลระหว่างตารางหลักและตารางรอง

28. บริษัทขนส่งข้ามชาติแห่งหนึ่งพัฒนาระบบจัดการฐานข้อมูลใหม่เพื่อจัดเก็บข้อมูลพนักงาน (Staff)
โดยกำหนดให้ 'Staff_ID' เป็น Primary Key และมีคอลัมน์ 'Passport_Number' เป็นคีย์สำรอง
ต่อมาบริษัทขยายสาขาไปยังประเทศที่สามซึ่งพนักงานบางส่วนไม่มีหนังสือเดินทาง เมื่อเจ้าหน้าที่
พยายามบันทึกข้อมูลโดยเว้นว่างช่อง 'Passport_Number' ระบบกลับปฏิเสธการบันทึกข้อมูล
ในขณะเดียวกัน เมื่อพนักงานลาออกและถูกลบข้อมูลจากตารางหลัก พบว่ารหัสพนักงานในตาราง
การเดินรถ (Logistics) ที่เคยอ้างถึงพนักงานคนนั้นกลับกลายเป็นค่าว่างทั้งหมดจนเสียประวัติ
จากสถานการณ์นี้นักศึกษาวิเคราะห์ว่าระบบกำลังเผชิญกับปัญหาการละเมิดกฎ Integrity ในส่วนใด
และแนวทางที่ถูกต้องในการจัดการกับพนักงานที่ไม่มีหนังสือเดินทางตามหลักการคือข้อใด

ก. ระบบละเมิดกฎ User-defined Integrity เนื่องจากพนักงานต่างชาติมีโครงสร้างข้อมูลต่างกัน; แนวทางแก้ไขคือการยกเลิกคีย์หลัก Staff_ID แล้วเปลี่ยนไปใช้ชื่อ-นามสกุลร่วมกันเป็น Composite Key เพื่อให้รองรับค่าว่างได้ตามหลักการของฐานข้อมูลเชิงสัมพันธ์ที่ยืดหยุ่น

ข. ระบบละเมิดกฎ Domain Integrity เพราะเลขพาสปอร์ตเป็นค่าว่าง และใช้วิธี NO ACTION ในการจัดการตารางลูก; แนวทางแก้ไขคือการสร้างตารางพนักงานแยกออกเป็นสองชุด คือพนักงานที่มีพาสปอร์ตและไม่มีพาสปอร์ต เพื่อให้สอดคล้องกับคุณสมบัติ Atomic Value ของรีเลชั่น

ค. ระบบละเมิดกฎ Entity Integrity ในคอลัมน์ Passport_Number และใช้วิธี SET NULL ในการลบพนักงาน; แนวทางแก้ไขคือควรกำหนดให้ Passport_Number ยอมรับค่า NULL ได้เพราะไม่ใช่ Primary Key และเปลี่ยนวิธีจัดการข้อมูลปลายทางเป็น RESTRICT เพื่อรักษาประวัติการทำงานไว้

ง. ระบบละเมิดกฎ Referential Integrity ในคอลัมน์ Staff_ID และใช้วิธี CASCADE ในการลบข้อมูล; แนวทางแก้ไขคือต้องบังคับให้ทุกคนมีเลขพาสปอร์ตโดยการสุ่มตัวเลขชั่วคราวใส่ลงไปเพื่อให้ผ่านกฎ Domain Integrity และรักษาความสอดคล้องของข้อมูลในระดับกายภาพ

29. สมมติว่าคุณกำลังออกแบบฐานข้อมูลให้กับร้านค้าปลีกที่มีหลายสาขา โดยมีตาราง
Products {ProductID, PName, Category, Price, Stock} และตาราง Branches{BranchID, Location, Manager}
ในการเชื่อมโยงข้อมูล คุณได้สร้างตารางกลางชื่อ 'Inventory_Link' เพื่อจัดเก็บข้อมูลสินค้าในแต่ละสาขา
โดยกำหนดให้ ProductID และ BranchID ร่วมกันเป็น Primary Key (Composite Key) ต่อมาเจ้าหน้าที่
พยายามเพิ่มรายการสินค้าใหม่โดยระบุเพียง ProductID และเว้นว่างช่อง BranchID ไว้เพื่อเติมทีหลัง
แต่ระบบ RDBMS ปฏิเสธการบันทึกข้อมูลทันที สถานการณ์นี้สะท้อนให้เห็นถึงข้อจำกัดในเรื่องใด
และเหตุใดระบบถึงไม่อนุญาตให้ดำเนินการแม้ว่า BranchID จะเป็นคีย์นอกที่อ้างอิงไปยังอีกตารางก็ตาม

ก. ละเมิดกฎ Referential Integrity เพราะระบบไม่สามารถตรวจสอบความสัมพันธ์ข้ามตารางได้หากคีย์ตัวใดตัวหนึ่งขาดหายไป ทำให้ข้อมูลในตาราง Inventory_Link กลายเป็นข้อมูลที่ไม่มีความสมบูรณ์เชิงกายภาพ

ข. ละเมิดกฎ Entity Integrity เพราะเมื่อใดก็ตามที่แอทตริบิวต์หนึ่งถูกกำหนดให้เป็นส่วนหนึ่งของคีย์หลัก แอทตริบิวต์นั้นจะห้ามมีค่าเป็น NULL โดยเด็ดขาดไม่ว่าจะทำหน้าที่เป็นคีย์นอกด้วยหรือไม่

ค. ละเมิดกฎ Domain Integrity เนื่องจากค่า NULL ไม่ถูกนับรวมอยู่ในเซตของข้อมูลชนิดตัวเลขที่ใช้เก็บรหัสสาขา ส่งผลให้การประมวลผลในระดับชั้น Conceptual Schema เกิดความผิดพลาดขึ้น

ง. ละเมิดกฎคุณสมบัติของทูเพิล (Tuple Uniqueness) เพราะการมีค่าว่างในคีย์หลักจะทำให้ระบบไม่สามารถคัดแยกความแตกต่างระหว่างสินค้าแต่ละชิ้นในตารางได้ ส่งผลให้ Cardinality ของรีเลชั่นลดลง

ยอดขายย้อนหลัง นักศึกษาวิเคราะห์ได้ว่าข้อกำหนดเกี่ยวกับ Discount_Rate และข้อกำหนด
การลบลูกค้า สอดคล้องกับคุณสมบัติใดของ Integrity Rules

ก. Discount_Rate คือ User-defined Integrity และการห้ามลบลูกค้าคือ Domain Integrity เพราะเกี่ยวข้องกับสถานะของลูกค้าในระบบจัดการรีเลชั่น

ข. Discount_Rate คือ Domain Integrity (กฎความถูกต้องของโดเมน) และการห้ามลบลูกค้าคือ Referential Integrity ในรูปแบบ RESTRICT (หรือ NO ACTION)

ค. Discount_Rate คือ Referential Integrity เพราะต้องอ้างอิงจากนโยบายบริษัท และการห้ามลบลูกค้าคือ Entity Integrity เพราะต้องรักษาความไม่ซ้ำกันของรายการขาย

ง. Discount_Rate คือ Entity Integrity (กฎความถูกต้องของเอนทิตี้) และการห้ามลบลูกค้าคือ Referential Integrity ในรูปแบบ CASCADE

31. จงพิจารณารีเลชั่น PROJECT_ASSIGN {Emp_ID, Proj_ID, Role, Join_Date} โดยกำหนดให้ Emp_ID และ Proj_ID
ร่วมกันทำหน้าที่เป็น Primary Key หากในอนาคตต้องการเพิ่มข้อมูลแต่พนักงานยังไม่ได้รับมอบหมายเข้าโปรเจกต์ใดเลย (Proj_ID ว่าง) หรือโปรเจกต์ยังไม่มีพนักงาน (Emp_ID ว่าง) แต่ผู้ใช้พยายามป้อนข้อมูลเพื่อจองสิทธิ์ไว้ก่อน ตามกฎความถูกต้อง ของเอนทิตี้ (Entity Integrity) การกระทำนี้สามารถทำได้หรือไม่ และส่งผลต่อโครงสร้างความสัมพันธ์อย่างไร

ก. ทำไม่ได้ เพราะกฎ Entity Integrity ระบุว่าส่วนประกอบใดๆ ของคีย์หลักจะว่างเป็น NULL ไม่ได้เด็ดขาด เพราะจะทำให้ระบบสูญเสียความสามารถในการระบุตัวตนของแต่ละทูเพิล

ข. ทำได้เฉพาะกรณีที่ Proj_ID ว่าง แต่ Emp_ID ห้ามว่าง เพราะพนักงานเป็นวัตถุหลัก (Entity) ส่วนโปรเจกต์เป็นเพียงกิจกรรมที่เกิดขึ้นตามความสัมพันธ์แบบ M:N

ค. ทำได้ หากเรากำหนดให้แอทตริบิวต์ที่ว่างนั้นเป็นคีย์นอก (Foreign Key) ไปด้วย เพราะกฎ Referential Integrity ยืดหยุ่นกว่ากฎความถูกต้องของเอนทิตี้ในเชิงปฏิบัติ

ง. ทำไม่ได้ เพราะจะไปขัดกับคุณสมบัติ Atomic Value ของรีเลชั่นที่ระบุว่าข้อมูลแต่ละช่องต้องเป็นหนึ่งเดียวและไม่สามารถเป็นค่าว่างได้ตามหลักการคณิตศาสตร์ของเซต

32. ในการออกแบบรีเลชั่น 'Inventory' ของห้างสรรพสินค้าแห่งหนึ่ง มีแอทตริบิวต์ดังนี้

ก. ไม่อนุญาต เพราะจะทำให้รีเลชั่นนี้สูญเสียความเป็นเอกภาพ (Entity Integrity) และทำให้ไม่สามารถคำนวณจำนวนสินค้าแยกตามผู้ผลิตได้แม่นยำ

ค. อนุญาตได้เฉพาะเมื่อเรากำหนดค่า Default ให้กับ Supplier_ID เป็นรหัสบริษัทตนเอง เพื่อป้องกันข้อมูลกำพร้าในระดับชั้น Physical Schema

ง. ไม่อนุญาตให้เป็น NULL เด็ดขาด เพราะคีย์นอกต้องมีค่าที่ปรากฏอยู่ในคีย์หลักของตารางแม่เสมอตามกฎ Referential Integrity ที่เคร่งครัด

33. กฎความถูกต้องของโดเมน (Domain Integrity) ไม่ได้ทำหน้าที่ตรวจสอบเพียงแค่ชนิดข้อมูล
